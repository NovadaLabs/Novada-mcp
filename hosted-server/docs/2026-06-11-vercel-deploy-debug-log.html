Seven Module-Init Bugs (all fixed)

Each bug crashed the function before any auth/dispatch code ran. Vercel hid them all behind a generic "FUNCTION_INVOCATION_FAILED" page. Only vercel logs <deployment-url> --no-branch --since 5m --expand revealed the actual stack traces.

#1 Missing vendor/package.json

commit 2b50312 · "Cannot find module '../package.json'"
What broke
vercel/vendor/novada-mcp/config.js does _require("../package.json") to read its version. The original 2026-06-02 vendor commit copied build/* but not the parent package.json. Module init throws.
Fix
Added minimal vercel/vendor/package.json with name + version fields.

#2 Stale vendored novada-mcp build

commit 83efcb7 · "does not provide an export named 'AiMonitorParamsSchema'"
What broke
Vendored tools/types.js was from before AiMonitorParamsSchema was added upstream. api/mcp.ts imports the new symbol but it doesn't exist in vendor.
Fix
rsync -a --delete from upstream ~/Projects/novada-mcp/build/. Now matches v0.8.0 (merge commit 839890c).
Side note
The prior HARD RULE about KR-6 staying local was lifted by upstream merge — confirmed safe to vendor.

#3 rsync clobbered browser stubs

commit c4079bd · "Cannot find package 'playwright-core'"
What broke
The 2026-06-02 deploy stubbed vendor/novada-mcp/utils/browser.js + 2 sibling files to drop playwright-core (the 50MB Hobby tier limit fix). Bug #2's rsync overwrote those stubs with the real playwright-importing code.
Fix
git checkout 36512bc -- <3 files> restored the stubs verbatim from the original stub commit.
Lesson
Future vendor-refresh: either --exclude the stubs or restore them immediately post-rsync.

#4 Missing type: "module" in vendor shim

commit e379584 · "does not provide an export named 'X'"
What broke
Even after re-vendoring, the same ESM-export error persisted. Diagnosed by running node --input-type=module -e 'import(...)' locally — Node warned: "Module type of types.js is not specified and it doesn't parse as CommonJS". The vendor package.json shim was missing "type": "module", so Node defaulted to CommonJS, which can't see ESM export statements.
Fix
Added "type": "module" to vercel/vendor/package.json.

#5 pdfjs-dist DOM globals missing in Node

commit 66447d8 · "ReferenceError: DOMMatrix is not defined"
What broke
Chain: api/mcp.ts → vendor/.../tools/extract.js → pdf-parse → pdfjs-dist → references DOMMatrix/ImageData/Path2D at module top-level. None exist in Node. First attempt put polyfills inline at top of api/mcp.ts, but ESM hoists import statements before any other code — so pdfjs crashed before the polyfill ran.
Fix
Extracted polyfills to separate vercel/api/_polyfills.js, imported as the very first side-effect import in api/mcp.ts. ESM resolves imports in source order; polyfills are in place before pdfjs loads.
Lesson
Inline polyfills don't work in ESM. They must be in a separately imported file listed first.

#6 new URL(request.url) fails on Node runtime

commit acfa798 · "TypeError: Invalid URL { input: '/mcp' }"
What broke
Vercel Edge: request.url is an absolute URL. Vercel Node: it's the path only ("/mcp"), per Node IncomingMessage convention. new URL("/mcp") requires a base.
Fix
Provide a base URL from the Host header: new URL(req.url, `https://${req.headers.host}`). Works in both runtimes — Edge ignores the base, Node uses it.

#7 Fetch API vs Node IncomingMessage

commit 6e93e5c · "request.headers.get is not a function"
What broke
The handler body was written for Fetch API (request.headers.get(...), new Response(...)) but Vercel Node runtime invokes with Node's IncomingMessage/ServerResponse. Different interface entirely.
Fix
Added a Node↔Fetch adapter at the boundary: default export is Node-style (req, res), converts to Fetch Request, calls the original handler (renamed fetchHandler), converts Response back to Node res.

#8 (remaining) MCP SDK transport: Fetch vs Node

PENDING · "Cannot read properties of undefined (reading 'headersSent')"
What's broken
Line 617-618 of api/mcp.ts: 
// @ts-expect-error — transport.handleRequest accepts Fetch Request/Response
//   when running in Workers/edge runtimes via the SDK's edge shim.
const response: Response = await transport.handleRequest(request);
The @ts-expect-error comment is from the original author: "I know this is type-wrong but I'm betting it works in an edge runtime." On Vercel Node, the MCP SDK's StreamableHTTPServerTransport.handleRequest expects Node-style (req, res) and writes the response stream directly to res. It does res.headersSent on undefined (because we passed only a Fetch Request).
Status
Function fully boots, receives requests, returns JSON-RPC envelopes. The MCP protocol dispatch is the only missing piece. Fix in progress.

Bonus Discovery — 5 silently-empty env vars

CLI vercel env add stdin bug for sensitive type

v53.2.0 · undocumented
Symptom
Running echo "value" | vercel env add NAME production --sensitive --force succeeds (exit 0, "Added env var") but the actual saved value is the empty string. vercel env pull confirms: NAME="".
Scope of damage
All 5 env vars set this way during project setup 8 days ago + 2 we set today were empty: KV_REST_API_URL, KV_REST_API_TOKEN, STUB_AUTH_WARNING_ACCEPTED, RATE_LIMIT_PER_MIN, FREE_PLAN_MONTHLY_QUOTA. The 6th (NOVADA_API_KEY) was set via the dashboard UI and was correct.
Fix
Bypassed CLI entirely — used Vercel REST API directly: 
DELETE /v9/projects/{id}/env/{envId}      # remove broken sensitive ones
POST   /v10/projects/{id}/env?upsert=true # create as type:encrypted
Verified via vercel env pull — values now non-empty. Script saved at ~/Projects/novada-mcpserver/scripts/ (no, not yet — TODO if needed).
Lesson
For Vercel env vars: use the dashboard UI or REST API. Avoid CLI's --sensitive flag entirely. The "sensitive" type is also worse operationally — values can't be retrieved even by admins, breaking vercel env pull workflows.

Lessons / Patterns Worth Remembering

L1 · Generic "FUNCTION_INVOCATION_FAILED" hides everything Always immediately get the deployment URL via API + run vercel logs <url> --no-follow --no-branch --since 5m --expand. The branch filter defaults to "main" but returns nothing if the function is in error state. --no-branch + the deployment URL gets the real stack trace.
L2 · ESM module init order is unforgiving import statements hoist to the top of the module BEFORE any other code runs. Polyfills, monkey-patches, env setup — all must be in a separately imported file listed FIRST in the import order.
L3 · Edge runtime ≠ Node runtime for request shape Code that works on Cloudflare Workers / Vercel Edge will break on Vercel Node Functions because request.url is path-only and request.headers is a plain object (no .get()). Either commit to one runtime end-to-end, or write a small adapter at the boundary.
L4 · Vendor refreshes need a checklist When rsync-ing upstream build/ into vercel/vendor/, three things must be preserved or re-applied: (1) browser-tool stubs that drop playwright-core, (2) vendor/package.json with type:module, (3) any environment-specific patches. Future vendor-refresh script should restore these post-rsync.
L5 · MCP SDK is Node-first; Fetch API requires explicit shim StreamableHTTPServerTransport.handleRequest defaults to Node (req, res) signature. The "edge shim" the original author referenced doesn't exist as of @modelcontextprotocol/sdk@^1.26.0. Either use Node objects end-to-end OR implement the JSON-RPC dispatch manually (bypass transport).
